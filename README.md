# Zoo ERP (SOLID demo, console)

## 1) Предметная область
Проект моделирует упрощённую ERP-подсистему зоопарка (консольное приложение).  
Система отражает базовые повседневные процессы:

- учёт животных: кличка, вид, вольер, рацион, расписание кормления, интересный факт;
- учёт сотрудников (упрощённо): имя + роль (FEEDER / VET);
- операции ухода:
  - кормление (с проверкой: “по расписанию” или “вне расписания”, хранится последняя запись),
  - медицинский осмотр при приёме (intakeExam — обязателен по регламенту),
  - лечение / процедура (treat — с заметкой);
- отчётный модуль:
  - общее количество животных,
  - группировка по видам и вольерам,
  - расписание кормления,
  - последние события по кормлению и медицине для каждого животного.

Цель проекта — показать архитектуру и применение SOLID, а не реализовать полноценную ERP.

---

## 2) SOLID — как реализовано

### 2.1 SRP — Single Responsibility Principle
- `Animal` и конкретные животные (`Lion`, `Parrot`, `Snake`) содержат только данные/описание и расписание кормления.
- `SimpleFeedingService` отвечает только за кормление и хранение последнего события кормления.
- `SimpleMedicalService` отвечает только за медицинские операции и хранение последней мед. записи.
- `ReportService` отвечает только за построение отчёта.

Итог: изменения в кормлении не требуют изменений в медицине/отчёте и наоборот.

---

### 2.2 OCP — Open/Closed Principle
Чтобы добавить новый вид животного, достаточно создать новый класс `class Bear implements Animal { ... }`.
Существующие сервисы не меняются, потому что работают через интерфейс `Animal`.

---

### 2.3 LSP — Liskov Substitution Principle
Любой объект, реализующий `Animal`, можно подставить вместо любого другого:
- сервисы кормления/медицины принимают `Animal`, а не конкретные классы,
- отчёт строится по списку `List<Animal>`, не завися от конкретных видов.

Контракт `Animal` задаёт единое поведение, которое обязаны корректно реализовать все животные.

---

### 2.4 ISP — Interface Segregation Principle
Вместо одного “толстого” интерфейса ухода сделаны два маленьких:
- `FeedingService` — только кормление и получение последнего события кормления,
- `MedicalService` — только мед. осмотр, лечение и получение последней мед. записи.

Это уменьшает лишние зависимости и упрощает развитие системы: можно добавлять новые сервисы (например уборку) отдельным интерфейсом.

---

### 2.5 DIP — Dependency Inversion Principle
`ReportService` зависит от абстракций (`FeedingService`, `MedicalService`), а не от конкретных реализаций.
Конкретные реализации (`SimpleFeedingService`, `SimpleMedicalService`) создаются в `main` и передаются в отчёт через конструктор.

Итог: можно заменить реализацию на более сложную (например с базой данных или логированием) без изменений кода отчёта.

---

## 3) Где появятся проблемы при расширении
Проект учебный, поэтому при росте требований будут ограничения:

1) Хранится только “последнее событие” (Map), а не история.  
   Если нужна история, придётся делать список событий или хранилище (БД).

2) Роли сотрудников заданы строками ("FEEDER"/"VET").  
   В крупном проекте лучше использовать enum или систему прав.

3) Нет планировщика по времени (cron/scheduler).  
   Сейчас лишь проверка “окно +-30 минут от расписания”.


---

## 4) Почему абстракции улучшили дизайн
- `Animal` отделяет бизнес-логику от конкретных видов животных: новый вид добавляется без изменений сервисов.
- Интерфейсы `FeedingService` и `MedicalService` позволяют менять реализацию без переписывания отчёта (DIP).
- Разделение на сервисы уменьшает связанность и упрощает поддержку (SRP/ISP).
- `ReportService` концентрируется только на отчёте и получает данные через интерфейсы.
